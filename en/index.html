    Overview

    iziData provides API interfaces such as identity verification, face recognition, handheld ID identification, ID card OCR, and personal/enterprise credit.

    iziData backend needs to verify sender’s identity of each request by using the Access Key / Secret Key encryption method. Access Key (AK) is used for identifying the user, Secret Key (SK) is the key used for user to encrypt the authentication string and for iziData backend to verify the authentication string, SK must be kept secret, only the user and iziData know.

    For SDK initialization, users need to apply their own AK and SK from iziData.

    Please make sure to replace YOUR_ACCESS_KEY and YOUR_SECRET_KEY with your own AK and SK.

    Please make sure to replace YOUR_ACCESS_KEY and YOUR_SECRET_KEY with your own AK and SK.

    Requests will return directly if the authentication fails.

    Response example for authentication fails.

    {
  "status": "[status code, refer to the following description]",
  "message": "error message"
}

    Timeout returned for non-Indonesia request

    {
    "error_msg":"connection or read data time out",
    "error_code":"SDK108"
}

    fields description
    status status code
    message cause of error

    Status code

    status description
    INSUFFICIENT_BALANCE insufficient balance
    INVALID_PARAMETER_ERROR request parameters invalid
    EXPIRED_REQUEST expired request, please try again
    NO_AUTHORIZATION authentication failed, user doesn't exist or invalid ak/sk
    RETRY_LATER system error

    liveness

    Live recognition process: mobile phone local action recognition + server anti remake recognition

    1. The Living SDK captures face information through the camera, and needs to do some simple living actions according to instructions, such as nodding, blinking, etc. If the action recognition is passed, you can proceed to the next step, otherwise it will not pass.
    2. After completing the above operations, the SDK will select a face image from the current stream as output, and the face image will be saved in the phone's settings directory.
    3. The face image needs to be sent to its own server to request the iziData anti remake interface for anti remake recognition, anti-copying recognition, and living body verification.

    SDK source code integration process (live Demo and SDK source code access, please contact the business to obtain)

    1. Copy the libs and assets files in the app to the corresponding main project category: com.offline.livedetection.face.FaceLiveActionManager.
    2. Living body detection function: faceTracker(), after the detection is successful, a face image will be generated and the face image will be used to request an anti remake interface for anti remake detection.
    3. In order to show intuitively in the demo, the anti-copying is carried out in the SDK, and the normal process needs to be executed on the server side.
