    概述

    iziData提供如身份证实名认证、人脸比对识别、手持身份证识别、身份证OCR、个人/企业征信等API接口。

    iziData服务后台需要通过使用Access Key / Secret Key加密的方法来验证某个请求的发送者身份。Access Key（AK）用于标示用户，Secret Key（SK）是用户用于加密认证字符串和iziData服务用来验证认证字符串的密钥，其中SK必须保密，只有用户和iziData知道。

    SDK初始化，需要用户先申请到属于自己的AK和SK。

    请确保将YOUR_ACCESS_KEY和YOUR_SECRET_KEY替换为你自己的AK和SK。

    请确保将YOUR_ACCESS_KEY和YOUR_SECRET_KEY替换为你自己的AK和SK。

    请求者身份验证不通过将直接返回

    身份验证不通过应答样例

    {
  "status": "[错误状态码，详见后面表格]",
  "message": "出错信息"
}

    非印尼机房请求发生的超时返回

    {
    "error_msg":"connection or read data time out",
    "error_code":"SDK108"
}

    返回数据格式

    参数 描述
    status 状态码
    message 错误原因

    状态码说明

    status 说明
    INSUFFICIENT_BALANCE 余额不足
    INVALID_PARAMETER_ERROR 请求参数不合法
    EXPIRED_REQUEST 过期请求，请重试
    NO_AUTHORIZATION 验证不通过，用户不存在或者校验信息有误
    RETRY_LATER 系统错误

    HTTP请求

    POST https://api.izi.credit/v2/identitycheck

    请求参数

    参数 描述
    name 可选，字符串，认证人员的姓名
    id 字符串，认证人员的身份证号NIK，格式为16位阿拉伯数字
    md5 可选，字符串，md5目前只支持印尼地区且姓名不支持md5，md5=1表示请求参数为32位md5

    返回数据格式

    参数 描述
    status 状态码
    message id：身份证号码，NIK
    name：姓名
    place_of_birth：出生地，可能为空
    date_of_birth：出生日期，可能为空
    gender：性别，可能为空
    address：居住地址，可能为空
    province：省份，可能为空
    city：城市，可能为空
    district：区县，可能为空
    village：村镇，可能为空
    rt：领组，可能为空
    rw：居委会，可能为空
    religion：宗教信仰，可能为空
    marital_status：婚姻状况，可能为空
    work：职业，可能为空
    nationnality：国籍，可能为空

    状态码说明

    status 说明
    OK 计费，查找成功，库中存在查询的NIK，且姓名比对一致
    NOT 计费，查找成功，库中存在查询的NIK，但姓名比对不一致
    PERSON_NOT_FOUND 免费，我们库中查不到此NIK
    INVALID_ID_NUMBER 免费，输入的NIK不合法
    NOT_FOUND_MD5 免费，请求的MD5不存在
    RETRY_LATER 免费，系统错误，请稍后重试

    活体识别

    活体识别流程：手机本地动作识别 + 服务端防翻拍识别

    1. 活体SDK通过摄像头捕获人脸信息，需要根据指令做一些简单的活体动作，比如点点头、眨眨眼等。如动作识别通过，可进行下一步操作，否则不通过。
    2. 在完成上述的操作之后，SDK会从视频流中选取出一帧人脸图像作为输出，人脸图片会保存在手机的设定目录中。
    3. 人脸图片上传到自己服务器中，用来请求iziData的防翻拍接口（请求文档如下），进行防翻拍识别，防翻拍识别通过，活体验证通过。

    SDK源码合入流程（活体Demo和接入SDK源码请联系商务获取）

    1. 将app中libs和assets文件拷贝到对应的主工程下类：com.offline.livedetection.face.FaceLiveActionManager。
    2. 活体检测函数：faceTracker()，检测成功后会生成人脸图像并用人脸图像请求防翻拍接口，进行防翻拍检测。
    3. 演示Demo中为了直观展示，防翻拍在SDK中进行，常规流程需要放在服务端中执行。
